Caching Service

Shwetha KamathApril 2026

What is Caching and Why is it Required?

Caching is the practice of storing frequently accessed data in a temporary, fast-access storage layer so that future requests can be served without repeatedly querying the primary data source. The cached copy sits closer to the consumer — either in memory or on a dedicated caching server — and delivers data significantly faster than a full database read.

Without caching, every single request hits the database. As traffic grows, this creates a bottleneck: queries queue up, response times increase, and the database eventually becomes the weakest link in the system. Caching addresses this by absorbing the majority of read traffic, keeping the database free for writes and complex queries.

Key reasons caching is required:

  • Reduced latency: Serving data from an in-memory cache (microseconds) is orders of magnitude faster than querying a database (milliseconds to seconds).
  • Lower database load: Fewer queries hit the primary database, reducing CPU usage, connection pressure, and the risk of outages under traffic spikes.
  • Cost efficiency: In cloud environments, database compute is expensive. Caching reduces the need for frequent scaling of database instances.
  • Improved availability: Even if the database experiences a brief outage, cached data can continue serving read requests, providing a buffer of resilience.
📌 When not to cacheCaching is most effective for read-heavy workloads with data that doesn't change on every request. For write-heavy or real-time-critical data (like live financial transactions), caching can introduce stale data risks and may not be the right approach.

Core Caching Strategies

1. Cache-Aside (Lazy Loading)

In the cache-aside pattern, the application is responsible for managing the cache. When a request comes in, the application first checks the cache. If the data is found (a cache hit), it's returned immediately. If not (a cache miss), the application queries the database, stores the result in the cache for future requests, and then returns the data.

The cache is populated lazily — data is only cached when it's actually requested. This means the cache naturally fills up with the most frequently accessed data over time.

Cache-Aside Pattern
function getProduct(productId) {
  // Step 1: Check the cache first
  const cached = cache.get(`product:${productId}`)
  if (cached) {
    // Cache hit — return immediately
    return cached
  }

  // Step 2: Cache miss — query the database
  const product = db.query(
    `SELECT * FROM products WHERE id = $1`, [productId]
  )

  // Step 3: Store in cache with a TTL (e.g., 5 minutes)
  cache.set(`product:${productId}`, product, { ttl: 300 })

  return product
}

Advantages: Only requested data gets cached (efficient memory usage), simple to implement, and the application retains full control over what enters the cache.

Disadvantages: The first request for any piece of data always hits the database (cold start penalty). Under high traffic with an empty cache, many simultaneous requests can storm the database — this is known as the thundering herd problem.

2. Write-Through

In the write-through pattern, every time data is written to the database, it is also simultaneously written to the cache. This ensures the cache is always up-to-date with the latest data, eliminating stale reads entirely.

Write-Through Pattern
function updateProduct(productId, data) {
  // Write to database
  db.query(
    `UPDATE products SET name=$1, price=$2 WHERE id=$3`,
    [data.name, data.price, productId]
  )

  // Simultaneously update the cache
  cache.set(`product:${productId}`, data)

  // Cache and DB are always in sync
}

Advantages: Cache is always consistent with the database. Read requests never encounter stale data. Ideal for use cases where data accuracy is critical — such as inventory counts, session data, or user preferences.

Disadvantages: Every write operation becomes slower because data must be written to two locations. Additionally, data that is rarely read still gets cached, which can waste cache memory.

💡 Which one to choose?Use Cache-Aside when your workload is read-heavy and slight staleness is acceptable (e.g., product catalogs, user profiles). Use Write-Through when data consistency between cache and database is a strict requirement (e.g., payment status, session tokens).

PostgreSQL as a Caching Layer with TTL

While dedicated caching systems like Redis or Memcached are the most common choices, PostgreSQL can also be used as a caching database — particularly when you want to keep your infrastructure simpler by avoiding additional services. PostgreSQL supports expiration-like behaviour through a TTL (Time-To-Live) pattern implemented at the application or table level.

How TTL Works with PostgreSQL

TTL defines how long a cached entry is considered valid. Each cached row stores a timestamp indicating when it was inserted or last refreshed. When a read request comes in, the application checks whether the entry has expired by comparing the current time against the stored timestamp plus the TTL duration.

PostgreSQL Cache Table
CREATE TABLE cache_store (
  cache_key    VARCHAR(255) PRIMARY KEY,
  cache_value  JSONB NOT NULL,
  created_at   TIMESTAMP DEFAULT NOW(),
  ttl_seconds  INTEGER DEFAULT 300  -- 5 minutes
);

-- Read with TTL check
SELECT cache_value FROM cache_store
WHERE cache_key = 'product:101'
  AND created_at + (ttl_seconds * INTERVAL '1 second') > NOW();

-- Periodic cleanup of expired entries
DELETE FROM cache_store
WHERE created_at + (ttl_seconds * INTERVAL '1 second') < NOW();

Why TTL is Required

  • Prevents stale data: Without TTL, cached data lives forever. If the source data changes, the cache serves outdated information indefinitely, leading to incorrect results.
  • Automatic memory management: TTL ensures old entries are cleaned up, preventing the cache table from growing unbounded and consuming excessive disk space.
  • Balances freshness and performance: TTL lets you control exactly how stale you're willing to let data get — 30 seconds for fast-changing data, 10 minutes for stable data.

Without TTL: The Fallback

If TTL is not implemented, cached data never expires. This leads to several problems:

  • Stale data served indefinitely: A product price updated in the source database will continue to show the old price from the cache until someone manually invalidates it.
  • Unbounded cache growth: The cache table grows continuously. Old, unused entries accumulate, consuming storage and slowing down queries on the cache table itself.
  • Manual invalidation burden: Without TTL, every code path that modifies source data must also explicitly delete or update the cache. Missing even one path creates inconsistencies that are difficult to debug.
📌 Best practiceAlways define a TTL, even if it's generous (e.g., 1 hour). It acts as a safety net — ensuring the cache self-corrects over time, even if explicit invalidation is missed somewhere in the code.

Complete Caching Architecture on AWS

The diagram below shows how caching fits into a typical AWS-hosted application stack. The request flows from the end user through multiple layers — each serving a specific purpose — before reaching the database.

End-to-end caching architecture — request flows from user through CDN, load balancer, API gateway (AppSync), application layer, then cache or database
Amazon CloudFront (CDN)A Content Delivery Network that caches static assets at edge locations globally, reducing latency by serving content from the server nearest to the user.
Application Load Balancer (ALB)Distributes incoming application traffic across multiple targets (servers, containers) based on request content such as URL path or host headers.
API Gateway / AppSyncA managed service that acts as the front door for APIs — handling request routing, authentication, throttling, and in the case of AppSync, serving GraphQL endpoints.
Amazon RDS (PostgreSQL)A managed relational database service that handles provisioning, patching, backups, and scaling of PostgreSQL instances in the cloud.

Data Retrieval with GraphQL via AWS AppSync

In this architecture, data is retrieved using GraphQL served through AWS AppSync. Instead of traditional REST endpoints, the client sends GraphQL queries that describe exactly what data is needed, and AppSync resolves those queries against the backend data sources (including the cache layer and the primary database).

What is GraphQL?

GraphQL is a query language for APIs developed by Facebook. Unlike REST, where each endpoint returns a fixed structure, GraphQL lets the client specify the exact fields it needs in a single request. The server returns only that data — nothing more, nothing less.

GraphQL Query Example
# Client requests only what it needs
query {
  product(id: "101") {
    name
    price
    category {
      name
    }
  }
}

# Response — no over-fetching, no under-fetching
{
  "data": {
    "product": {
      "name": "Wireless Headphones",
      "price": 2499,
      "category": { "name": "Electronics" }
    }
  }
}

Why GraphQL?

  • No over-fetching: REST endpoints often return entire objects even when the client needs only two fields. GraphQL eliminates this waste — the client gets precisely what it asks for, reducing payload size and bandwidth.
  • No under-fetching: In REST, fetching related data (e.g., a product and its category) often requires multiple API calls to different endpoints. GraphQL resolves nested relationships in a single query.
  • Single endpoint: Instead of managing dozens of REST endpoints (/products, /products/:id, /categories), GraphQL operates through one endpoint. The query itself defines what data to return.
  • Strongly typed schema: GraphQL enforces a schema that defines every type, field, and relationship. This serves as a contract between frontend and backend teams, reducing miscommunication and integration errors.
  • Efficient for mobile clients: Mobile apps on slower networks benefit significantly from receiving only the data they need, reducing load times and data consumption.

Benefits of AWS AppSync

AWS AppSync is a fully managed GraphQL service that removes the operational burden of running your own GraphQL server. Key benefits include:

  • Managed infrastructure: No servers to provision or scale — AppSync automatically handles traffic spikes.
  • Built-in caching: AppSync supports server-side caching at the resolver level, which pairs well with the PostgreSQL caching strategy described above.
  • Real-time subscriptions: AppSync natively supports WebSocket-based subscriptions, allowing clients to receive live updates when data changes.
  • Multiple data source integration: A single AppSync API can connect to DynamoDB, RDS (PostgreSQL), Lambda functions, HTTP endpoints, and more — making it flexible for complex architectures.
  • Fine-grained authorization: AppSync integrates with Cognito, IAM, and API keys, giving precise control over who can query or mutate specific data.
💡 GraphQL + CachingWhen AppSync resolves a query, the resolver can first check the PostgreSQL cache table (with TTL). On a cache hit, the response is immediate. On a miss, the resolver queries the primary database, populates the cache, and returns the data — implementing the cache-aside pattern at the resolver level.
